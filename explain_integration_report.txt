Explain integration report
==========================

Fecha: 2026-06-20
Rama: feature/web-frontend

Objetivo implementado
---------------------
Se implemento unicamente la integracion funcional con:

POST ${NEXT_PUBLIC_API_URL}/explain

La llamada a /explain se ejecuta solo despues de una respuesta exitosa de /predict y reutiliza exactamente el mismo payload enviado a /predict.

No se implemento:
- GET /metrics
- gauge visual
- barra de longevidad definitiva
- diseno final
- visualizacion SHAP final

Archivos creados
----------------
- explain_integration_report.txt

Archivos modificados
--------------------
- web/components/SchemaForm.tsx
- web/lib/api.ts

Payload reutilizado
-------------------
El frontend reutiliza el mismo objeto payload que construye para /predict:

{
  "features": {
    "CODIGO": number | string | null
  },
  "edad_cronologica": number | null
}

Confirmacion de contrato:
- /predict recibe PredictRequest en api/schema.py.
- /explain tambien recibe PredictRequest en api/main.py.
- Por eso ambos endpoints comparten el mismo payload.
- edad_cronologica viaja fuera de features.
- features contiene solo codigos de schema.features.

Estructura exacta de respuesta /explain
---------------------------------------
Confirmada desde api/model_registry.py:

{
  "base_value": number,
  "contribuciones": [
    {
      "feature": string,
      "shap": number,
      "empuja": "longevo" | "no_longevo"
    }
  ]
}

Notas:
- contribuciones viene ordenado por mayor impacto absoluto.
- El backend usa top_n=8 por defecto.
- empuja es "longevo" cuando shap > 0.
- empuja es "no_longevo" cuando shap <= 0.

Implementacion frontend
-----------------------
- web/lib/api.ts ahora exporta EXPLAIN_URL.
- web/components/SchemaForm.tsx:
  - llama POST /predict al enviar el formulario.
  - si /predict responde OK, guarda y muestra el JSON temporal de /predict.
  - despues llama POST /explain con el mismo payload.
  - muestra estado "loading explain".
  - muestra estado "explain success".
  - muestra estado "explain error" si /explain falla.
  - si /explain falla, no borra ni rompe la visualizacion de /predict.
  - renderiza contribuciones en una tabla simple con columnas:
    - feature
    - shap
    - empuja
  - empuja="longevo" se muestra con indicador positivo verde.
  - empuja="no_longevo" se muestra con indicador negativo rojo.

Verificacion frontend
---------------------
Comando ejecutado:
cd web
npm.cmd run build

Resultado:
- Next.js compilo correctamente.
- TypeScript paso correctamente.

Como probarlo
-------------
1. Levantar API:
   pip install -r api/requirements.txt
   uvicorn api.main:app --reload

2. Asegurar modelos disponibles:
   kedro run --pipeline nhanes_2015
   kedro run --pipeline serving

3. Levantar frontend:
   cd web
   npm install
   npm run dev

   En PowerShell, si npm.ps1 esta bloqueado:
   npm.cmd install
   npm.cmd run dev

4. Abrir:
   http://localhost:3000

5. Completar campos requeridos y enviar.

Resultado esperado:
- /predict responde y se muestra su JSON.
- Luego aparece "loading explain".
- Si /explain responde OK, aparece "explain success" y una tabla con feature, shap y empuja.
- Si /explain falla, aparece un mensaje de error de explain sin borrar el resultado de /predict.

Confirmacion final
------------------
La integracion /explain reutiliza exactamente el mismo payload que /predict y coincide con el contrato del backend.
